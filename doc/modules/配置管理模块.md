# 配置管理模块

> 返回 [项目架构深度分析](../项目架构深度分析.md)

---

## 一、模块定位

`config/` 包负责 OpenCode 配置文件的读写操作。核心为 `oh-my-openagent.jsonc`（模型配置）和 `opencode.jsonc`（供应商配置）。

---

## 二、文件结构

| 文件 | 职责 |
|------|------|
| `model_config.go` | oh-my-openagent.jsonc 的行级 model 值替换 |
| `agent_descriptions.go` | agent/category 描述表读写 (`agents-comments.json`) |
| `provider_config.go` | opencode.jsonc 的供应商配置读写 |
| `scheme_config.go` | 方案目录路径配置 |

---

## 三、模型配置（model_config.go）

### 3.1 配置文件

- **路径**：`~/.config/opencode/oh-my-openagent.jsonc`
- **结构**：按 agent / category 分组，每项包含 model 字段

```jsonc
{
    "agents": {
        "sisyphus": { "model": "deepseek/deepseek-v4-pro" },
        "oracle": { "model": "deepseek/deepseek-v4-pro" }
    },
    "categories": {
        "quick": { "model": "opencode/big-pickle" },
        "visual-engineering": { "model": "opencode/big-pickle" }
    }
}
```

### 3.2 核心功能

| 操作 | 方法 | 说明 |
|------|------|------|
| 读取 | `LoadConfig()` | 解析 JSONC → OpenAgentConfig + 原始文本 |
| 保存 | `SaveConfig(entries)` | 逐行遍历原始文本，正则替换 model 值 |
| 新增条目 | `AddConfigEntry(key, model, type)` | 在指定 section 末尾插入新条目 |
| 删除条目 | `DeleteConfigEntry(key, type)` | 精确删除指定 key 的对象块 |
| 新增类型 | `AddModelType(type)` | 在根对象末尾插入新的空 section |
| 删除类型 | `DeleteModelType(type)` | 删除整个 section 块 |
| 全量读写 | `GetFullConfig()` / `SaveFullConfig(json)` | 前端直接读写完整 JSON 字符串 |
| 转换 | `ConfigToEntries(config, descriptions)` | 将 Go 结构体转为前端 ModelEntry 列表 |

### 3.3 保存策略

`SaveConfig` 采用**行级精确编辑**，不序列化整个 JSON 再写回：

1. 读取原始文件，按行拆分
2. 对每个 entry，在对应 section 内定位 `"model"` 行，正则替换值
3. 新增 / 删除条目时，在 section 内插入 / 移除对应的行块
4. 不再写入行尾注释（描述来自 `agents-comments.json`）

### 3.4 原子写入

```go
func writeConfigFile(path string, data []byte, perm os.FileMode) error {
    // 1. 验证 JSON 有效性
    // 2. 写入临时文件（同目录，确保原子 rename）
    // 3. Sync + Chmod + Close
    // 4. os.Rename 替换原文件
}
```

防止崩溃损坏配置文件。

## 四、描述管理（agent_descriptions.go）

### 4.1 描述文件

- **路径**：`exeDir/configs/oh-my-openagent/agents-comments.json`
- **格式**：`{ "agent_name": "描述文本", ... }`
- **部署**：构建时复制到 `build/bin/configs/oh-my-openagent/`

```json
{
  "oracle": "只读高级顾问：架构决策、疑难调试、代码审查，必须使用最强推理模型",
  "quick": "小修、错别字、单文件简单改动：优先低延迟和低成本"
}
```

### 4.2 核心功能

| 方法 | 说明 |
|------|------|
| `LoadAgentDescriptions()` | 读取描述文件，返回 `map[string]string` |
| `ApplyDescriptions(entries)` | 合并写入描述文件，保留已有条目 |

### 4.3 与保存联动

`app.go` 的 `UpdateModels()` 在保存模型配置后，自动调用 `ApplyDescriptions()` 将编辑中的 key-comment 合并写入描述文件。这意味着用户在 OMO 页面添加条目时填入的描述，会在保存后持久化。

## 五、供应商配置（provider_config.go）

### 5.1 配置文件

- **路径**：`~/.config/opencode/opencode.jsonc`
- **结构**：provider 字典 + enabled_providers 列表

### 5.2 核心功能

| 操作 | 方法 | 说明 |
|------|------|------|
| 读取 | `GetProviders()` | 解析 → ProviderInfo 列表 |
| 保存 | `SaveProvider(ps)` | 新增或更新单个供应商 |
| 删除 | `DeleteProvider(key)` | 删除供应商 |
| 路径 | `OpenCodeConfigPath()` | 返回配置文件路径 |

### 5.3 设计特点

- 供应商配置使用完整的 JSON 序列化 / 反序列化（与模型配置的行级编辑不同）
- `SaveProvider` 自动管理 `npm`、`setCacheKey` 等默认字段
- `Enabled` 通过 `enabled_providers` 列表管理

---

## 六、两者对比

| 维度 | 模型配置 | 供应商配置 |
|------|---------|-----------|
| 文件 | oh-my-openagent.jsonc | opencode.jsonc |
| 编辑策略 | 行级 model 值替换，不保留行尾注释 | 完整序列化/反序列化 |
| 注释来源 | 外部描述文件 `agents-comments.json` | 结构固定，注释较少 |

## 七、本地方案管理

### 7.1 目录与文件

- **方案目录**：`.sisyphus/omo-schemes/`（`service/scheme.go`）
- **文件格式**：`.jsonc`
- **命名**：用户自定义

### 7.2 核心操作

| 操作 | 前端 | 后端 |
|------|------|------|
| 列表 | `api.ListSchemes()` | `ListSchemes()` — 扫描方案目录 |
| 读取 | `api.ReadScheme(name)` | `ReadScheme(name)` — 读取文件内容 |
| 入库 | `api.SaveScheme(name, content)` | `SaveScheme(name, content)` — 写入方案目录 |
| 导出 | `api.ExportConfig(dir, name, content)` | `ExportConfig(dir, name, content)` — 写入用户目录 |
| 打开 | `api.OpenSchemeDir()` | `OpenSchemeDir()` — 文件资源管理器 |
