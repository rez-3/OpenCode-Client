## 三、发现的问题（按严重程度排列）

### 🔴 严重

#### 1. API 密钥明文泄露到前端 (`provider.go:40-139`)

**问题描述**：

```go
// provider.go line 40: 注释说"脱敏显示后4位"，但实际代码没有做任何脱敏
ApiKey  string `json:"apiKey"`  // API Key（脱敏显示后4位）

// line 118-119: 从配置文件读取完整的 API Key，直接传给前端
if v, ok := entry.Options["apiKey"].(string); ok {
    apiKey = v
}
// line 138: 原样返回
ApiKey:  apiKey,
```

**风险**：所有供应商的 API 密钥无脱敏地通过 Wails 绑定暴露给前端 JavaScript，一旦前端有 XSS 漏洞或被恶意插件注入，密钥将完全泄露。`struct` 注释明确标注了要脱敏但功能未实现，这是一个明显的遗漏。

**建议修复**：在 `getProviders()` 中添加脱敏函数：
```go
func maskAPIKey(key string) string {
    if len(key) <= 4 {
        return "****"
    }
    return strings.Repeat("*", len(key)-4) + key[len(key)-4:]
}
```

### 🟡 中等

#### 2. `config.go` JSONC 注释剥离逻辑存在边缘情况 (`stripComments`, line 525-541)

**问题描述**：

```go
func stripComments(text string) string {
    for _, line := range lines {
        idx := strings.Index(line, "//")
        if idx >= 0 {
            before := line[:idx]
            if strings.Count(before, "\"")%2 == 0 {  // ← 问题点
                line = before
            }
        }
    }
}
```

当前逻辑假设 `"` 总是成对出现才判定 `//` 在引号外。但在以下边缘情况可能出错：
- 字符串值中包含转义引号 `\"`（计数会偏移）
- 多行字符串（JSON 本身不支持，但 JSONC 格式被用户手动编辑时可能出现）

**影响**：低概率导致配置解析失败。

#### 3. `web.go:363` JSON 解析错误被静默忽略

**问题描述**：
```go
json.Unmarshal([]byte(knownDirs), &extraDirs)
```

如果前端传入的 `knownDirs` JSON 格式异常，静默失败后 `extraDirs` 为空切片，导致 global 项目下看不到已创建的会话，且没有任何错误反馈。

**建议修复**：添加错误日志：
```go
if err := json.Unmarshal([]byte(knownDirs), &extraDirs); err != nil {
    log.Printf("[WARN] 解析 knownDirs 失败: %v", err)
}
```

#### 4. 前端 `main.js` 3162 行单文件

虽然 JS 本身语法没问题，但单文件 3162 行没有任何模块化——主题切换、Wails API mock、会话管理、消息渲染、项目树、技能管理全部混在一起。长期维护成本高。

#### 5. 测试覆盖不均衡

| 源文件 | 测试文件 | 测试数 |
|--------|---------|--------|
| `config.go` | ✅ `config_test.go` | 9 个 |
| `web.go` | ✅ `web_test.go` | 3 个 |
| `skills.go` | ❌ | 0 |
| `symlink.go` | ❌ | 0 |
| `provider.go` | ❌ | 0 |
| `models.go` | ❌ | 0 |

技能管理（toggle、批量操作）和符号链接创建的核心逻辑完全没有自动化测试。

---

### 🟢 低

#### 6. `go.mod` 使用不存在的 Go 版本

```
go 1.25
```

当前 Go 稳定版是 1.24.x，`go 1.25` 是一个未来版本号。在标准 `go mod tidy` 工具链中可能产生兼容性警告。

#### 7. `skills.go` YAML frontmatter 解析过于简化 (`extractFrontmatterField`, line 115-146)

只支持单行 `field: value` 格式，不支持：
- 多行值（折叠/块标量）
- 嵌套 YAML
- 转义字符
- BOM 或前导空行

对于当前使用场景（SKILL.md 的 `name` 和 `description` 字段）来说问题不大，但如果有人写复杂 frontmatter 会解析失败。

#### 8. Goroutine 错误信息被丢弃 (`web.go:169`)

```go
go func() {
    _ = cmd.Wait()  // ← 进程崩溃无日志
}()
```

如果 opencode serve 意外崩溃，没有任何日志记录原因。

#### 9. `symlink.go` 链接验证不检查目标正确性 (`isLinked`, line 114-134)

如果符号链接存在但指向错误位置（如被手动修改），`isLinked` 仍然返回 `true`。

#### 10. `config.go:489` 临时文件清理依赖 `defer`

```go
defer os.Remove(tmpPath)
```

`os.Rename` 成功后，`Remove` 对已移动的文件失败（无害），但每次写配置都会有两次无用的文件系统调用。

## 四、正面评价（设计亮点）

1. **原子写配置**（`config.go:475-511`）：`writeConfigFile` 使用 "写临时文件 → 校验 → rename" 模式，有效防止配置损坏。

2. **写前校验**：`validateJSONC` 确保不会写入无效 JSONC，即使内部处理出错也不会破坏配置文件。

3. **锁设计合理**：`configWriteMu` (Mutex) 用于写入串行化，`cachedModelsMu` (RWMutex) 支持读并发，读写分离正确。

4. **Windows 符号链接回退**（`symlink.go:16-35`）：`os.Symlink` → `mklink /J` 的二级降级策略，兼顾开发模式和非管理员场景。

5. **进程树清理**（`web.go:620-623`）：`taskkill /F /T` 确保子进程（bun）不会被遗留。

6. **SSE 转发机制**（`web.go:511-557`）：通过 Wails Events 桥接 OpenCode SSE 到前端，实现流式输出。

7. **防御性编程**：多处空值检查、类型断言安全检查（如 `isReparsePoint` 使用 `os.Readlink`）。

8. **配置注释保留**（`config.go:691-716`）：`saveConfig` 通过行级操作保留 JSONC 中的 `//` 注释，避免配置被 `MarshalIndent` 覆盖后丢失人工标注。

## 五、建议修复优先级

| 优先级 | 问题 | 建议修复方案 |
|--------|------|-------------|
| **P0** | API 密钥未脱敏 | 在 `getProviders()` 中对 `ApiKey` 做 `maskAPIKey()` 处理（保留后4位） |
| **P1** | JSON 解析错误被忽略 | 对 `json.Unmarshal` 返回的 `err` 做 `log.Printf` 日志记录 |
| **P1** | 增加测试覆盖 | 为 `skills.go` 和 `symlink.go` 添加核心逻辑的单元测试 |
| **P2** | 前端代码拆分 | 将 `main.js` 按视图模块拆分为多个文件（如 `chat.js`, `skills.js`, `tree.js`） |
| **P3** | Go 版本号修正 | `go.mod` 中改为 `go 1.24` 或实际使用的版本 |
| **P3** | Goroutine 错误日志 | 对 `cmd.Wait()` 的返回值增加 `log.Printf` |
| **P3** | 前端 YAML 解析加强 | 如果遇到复杂 frontmatter 需求，引入轻量级 YAML 解析 |

## 六、总结

这个项目是一个**功能完整、可工作的桌面工具**，核心逻辑（符号链接管理、JSONC 配置读写、进程管理）实现正确，构建和测试全部通过。

**最需要立即关注的问题**：API 密钥未脱敏（P0 安全风险）。

其余问题属于代码质量和工程健康度层面，不影响当前功能，但会随着项目规模增长逐渐成为维护负担。整体代码质量在桌面工具项目中处于中上水平，架构简单直接，锁机制和原子写等并发安全问题处理得当。
