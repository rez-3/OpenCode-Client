## 一、分层架构

```text
前端（WebView2 / 浏览器）
  ├─ 工作区 / 供应商配置 / OMO 配置 / 技能管理 / 常用命令
  ├─ SSE 消息缓存与增量渲染
  ├─ 技能文件浏览器 / Web 目录浏览器
  └─ 手机端响应式布局与长会话窗口化

Wails Runtime Bridge / 页面 Web app-call RPC

Go 后端（App 门面）
  ├─ config/   配置读写
  ├─ service/  进程、前端 Web 服务、目录浏览器、SSE、树与 API 代理
  ├─ skill/    技能扫描、链接管理、文件浏览与保存
  └─ model/    类型定义
```

---

## 二、关键设计决策

1. **零框架前端**：纯原生 JS，按依赖顺序加载，无需构建工具
2. **通用 API 代理**：`OpenCodeAPI(method, path, body)` 不限制端点
3. **SSE 透传**：Go 只转发事件，消息缓存与渲染主要由前端控制
4. **双模式运行**：Wails 桌面模式 + 浏览器页面 Web 模式
5. **app-call RPC**：页面 Web 通过统一 RPC 访问 `App` 门面方法
6. **技能打开统一化**：桌面端与 Web 端都使用站内文件浏览/编辑器，而不是资源管理器
7. **移动端工作区专用策略**：项目树抽屉、隐藏工具栏、长会话窗口化

## 三、当前演进重点

当前项目已从单一桌面管理器演进为同时覆盖：

- 桌面端工作流
- 浏览器访问工作流
- 移动端工作区工作流

因此架构上的重点不再只是“进程管理”，而是：

- **能力桥接**：桌面与 Web 统一数据流，但交互差异化
- **文件浏览安全边界**：技能目录与 Web 目录浏览器都必须限制访问范围
- **前端性能控制**：长会话、移动端布局、SSE 增量更新需要避免全量重绘
