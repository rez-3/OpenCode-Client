## 二、关键设计决策

1. **零框架前端**：纯原生 JS，12 个文件按依赖顺序加载，无需构建工具
2. **通用 API 代理**：`OpenCodeAPI(method, path, body)` 不限制端点，前端可调用任意 OpenCode API
3. **SSE 透传**：Go 只做解析转发，事件分发、消息缓存和界面更新主要由前端 `chat.js` 负责
4. **双模式运行**：`api-mock.js` Proxy 模式，前端可脱离 Go 独立在浏览器预览
5. **行级配置编辑**：`SaveConfig` 逐行替换 model 值，不重建整段，保留文件结构
6. **原子写入**：先写临时文件 → rename，防止崩溃损坏配置
7. **进程树清理**：`taskkill /F /T` 确保子进程（bun）也终止
8. **描述文件分离**：agent/category 描述存独立 JSON，与配置文件解耦，保存时自动同步

## 三、前端双模式

```javascript
// api-mock.js — Proxy 自动路由
const api = new Proxy({}, {
    get(_, prop) {
        if (window.go && window.go.main.App[prop]) {
            return window.go.main.App[prop];  // Wails 真实 API
        }
        return mockApi[prop];  // 浏览器 Mock 回退
    }
});
```

同一套前端代码，Wails 桌面模式调用 Go 方法，浏览器模式使用 Mock 数据。

## 四、核心数据流

### SSE 事件流
```
OpenCode serve → /global/event SSE
  → service/sse.go 逐行解析
  → wruntime.EventsEmit("oc-event", data)
  → 前端 chat.js 按 type 分发
  → 消息区 / 右侧面板 / 项目树按需更新
```

当前实现中，**消息流增量更新**与**项目树刷新**是分离的：

- `message.updated` / `message.part.updated` / `message.part.delta` 这类高频事件只更新消息缓存和消息区 DOM
- `session.status` / `session.idle` 主要更新会话 busy/idle 状态、发送按钮和消息加载
- 项目树只在显式主会话创建命中、`session.deleted` 或其他显式业务动作（如新建、删除、添加目录、手动刷新）时全量重建；回复过程中由子任务派生出的 `session.created` 不会直接触发树刷新

### 模型配置流
```
oh-my-openagent.jsonc
  → GetFullConfig() → 前端 parse → modelEntries (UI)
  → 编辑 → 同步 workingConfigJson
  → 保存 → UpdateModels() → SaveConfig (行级) + ApplyDescriptions (描述文件)
```

### 方案管理流
```
.sisyphus/omo-schemes/
  → ListSchemes() → 下拉框
  → 选择 → ReadScheme() → workingConfigJson → modelEntries
  → originalEntries 保持系统基线 → 差异自动检测
  → 保存 → 写回系统配置 + 同步描述
  → 入库/导出 → workingConfigJson 完整结构
```
